The product

One marketplace, five product types, one pipeline

Clay sells procedural game assets, environment kits, procedural tools, courses, and bundles through a single publishing and ownership pipeline. A few architectural decisions hold the whole thing together:

core Ownership is one source of truth

  • Every "who owns what" question is answered by a single Entitlement row per buyer + product — whether it came from a purchase, a bundle, a manual grant, or a membership.
  • Library, downloads, course access, and checkout dedupe all read from the same place, so they never disagree.

core Money is exact

  • All prices are stored as integer cents + currency — never floating point — so totals, discounts, and invoices are penny-accurate.
  • Orders and invoices freeze a snapshot at purchase time, so historical records never drift when prices later change.

core Courses are products

  • A course is the same Product as an asset, so it flows through the same admin, pricing, cart, and checkout. Buying one auto-enrolls the student.

core Bundles fan out

  • Buying a bundle grants individual ownership of every item inside it — each lands in the library separately, ready to download.
  • The same fan-out powers memberships: a tier that includes a bundle gives subscribers all its members.

What's built

The 13 feature areas

Each area below is fully implemented in code. Expand any one for detail.

① Storefront & catalog›

Faceted browse & search

  • Filter by type, category, keyword, industry, theme, style, and price, with live facet counts that update with the query.
  • Trigram-accelerated search; the AI path adds ranked full-text + semantic search.

Rich product pages

  • Media gallery, spec sheet pulled from typed per-category data, license, and sale-aware pricing.
  • Procedural tools show an editable-parameter table (parameter / type / range / default).

Generative blueprint art

  • Every product without a photo gets a deterministic, seeded SVG "blueprint" cover unique to its type — wireframe, contour map, node graph, orbit diagram, or stacked frames.

Before / after slider

  • A draggable, keyboard-accessible wipe between "Clay" (the blueprint) and "Shipped" (the final render) — the brand story in one interaction.
② Live 3D configurator›

Real-time parametric model

  • A three.js / react-three-fiber scene on the landing page rebuilds a construction tank live as you drag sliders — drag to orbit, auto-rotate, soft shadows, three-point lighting.

Eight live parameters

  • Height, platforms on/off, floor count, side rails, ladder rungs, ladder tilt, weathering, and finish (concrete / galvanized / painted / weathered steel).
  • Parameter names mirror a real Houdini Digital Asset 1:1, so the same controls could later drive a server-side cook.
③ AI concierge "Atlas"›

Real Claude, with tools

  • Powered by Claude (Sonnet), streaming, able to call tools to search the catalog, pull product details, and gather web context.
  • Recommendations are real Clay products only — it never sends shoppers off-platform.

Hybrid search under the hood

  • Three tiers with graceful fallback: semantic (vector similarity via Voyage embeddings) → full-text → keyword. Filters always respected.

Inline product cards

  • Results render as real storefront cards inside the chat. Entry points: a landing "find assets" prompt and a persistent floating "Ask Atlas" panel.

Logged for insight

  • Every conversation is logged with what it recommended — feeding the admin "interest & catalog-gap" analytics.
④ Cart, checkout & payments›

Membership-aware cart

  • Cart lives in a cookie, self-heals dropped items, and applies the membership lens — tier-included items show as free, others get the member discount.

Safe checkout

  • Removes already-owned and covered items, enforces a single currency, creates a frozen-snapshot order, and opens a provider checkout. Ownership is never granted here.

Provider-agnostic

  • A clean payments interface with a fully working Stripe adapter; switching providers is a config change, not a rewrite.
⑤ Webhooks → entitlements → invoices → receipts›

One trustworthy grant point

  • Ownership is granted only by the verified payment webhook, and the whole pipeline is idempotent — replays never double-grant or double-invoice.

Bundles & courses handled

  • Bundle purchases fan out to every member; course purchases auto-create an enrollment.

Invoices & receipts

  • Sequential numbered invoices (INV-000042) rendered as PDF on demand; an HTML receipt email on every paid order.

Refunds

  • A refund revokes download access while leaving membership/grant access intact; re-purchase restores it.
⑥ Secure downloads›

Ownership-gated, expiring links

  • Every download passes one choke point: existence → ownership → file ready → rate limit → a 5-minute signed link. Non-owners get a clean 403.

Audited & throttled

  • Each download is logged (who, what version, when, expiry); a per-user rate limit guards against abuse.
⑦ Courses & learning platform›

Course builder

  • Build chapters, lessons, and quizzes; reorder freely; edit answer keys; set passing score, retakes, and "required to continue".

Protected video

  • Unlisted-YouTube playback through a privacy player — only the video id is stored, never the raw URL, and only entitled students reach it.
  • Watch progress is tracked; a lesson completes at 95% watched.

Real quizzes

  • Four question types (single / multi choice, true-false, short answer), all graded server-side so answers can't leak. Attempts, scores, and retakes are tracked.
⑧ Bundles›

Bundle management

  • Add/remove members with guards (no nesting, no self-inclusion). The product page shows the savings vs. buying separately.
⑨ Memberships (Free / Pro / Studio)›

Three tiers

  • Free à-la-carte · Pro 15% member discount + included items free · Studio 30% discount, includes everything Pro does.

Self-healing access

  • Membership access is reconciled automatically whenever a tier, a product's inclusion, or a bundle's contents change — and it's bundle-aware.
⑩ Reviews›

Verified-owner reviews

  • Only buyers who own a product can review it; 1–5 stars + optional text, editable. The product page shows the average and recent reviews without slowing the cached page.
⑪ Admin back office›

Dynamic product CRUD

  • One form that changes its fields by product type, with validation, slug-collision checks, search re-indexing, and draft → publish workflow.

Analytics dashboard

  • Revenue, orders by status, top sellers, catalog matrix, ownership counts, audience, downloads, average rating, and Atlas concierge metrics (sessions, match rate, catalog gaps).

Orders ledger & course builder

  • Read-only order history plus the full course authoring surface.
⑫ Storage layer (R2 / S3)›

Selectable backend

  • Cloudflare R2 or AWS S3, chosen by config, behind one interface — resumable multipart upload, signed downloads, listing, and server-side copy.

Import by reference

  • The admin console can browse an existing bucket and register objects as products/versions without re-uploading — ideal for the studio's existing asset library.
⑬ Accounts, roles & safety›

Roles

  • Buyer / Editor / Admin with revocable database sessions and guards on every protected page and API.

Rate limiting

  • Download issuance is throttled per user (and fails open if the limiter is unavailable, so genuine buyers are never blocked).

Straight talk

What's live, and what's intentionally pending

The marketplace is implemented end-to-end. A few things are deliberately deferred or wait on an API key — listed honestly so there are no surprises in the demo.

!
Interactive login isn't wired yet. The entire roles/sessions/gating system is built; only the sign-in providers (magic-link, OAuth) are deferred. A demo-only "sign in as editor" exists for walkthroughs and is hard-blocked in production. For the art-team demo, the public storefront, landing, configurator, pricing, and product pages are all fully browsable without login.
!
Integrations degrade gracefully without keys. No keys, no crash. Atlas falls back to catalog search without an Anthropic key; semantic search downgrades to full-text without a Voyage key; receipts log instead of send without Resend; rate limiting and storage simply report "not configured." Add a key and the feature lights up.
!
Paddle is a stub; Stripe is complete. The second payment provider is scaffolded but not implemented — Stripe is the working path.
!
Asset checksums & YouTube duration auto-fetch are modeled but not yet wired to a background job / API key (duration can be entered by hand).
